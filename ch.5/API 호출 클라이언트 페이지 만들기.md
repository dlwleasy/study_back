# 스프링 부트 애플리케이션 실행하기&#x20;


## 1. 스프링 프레임워크와 스프링 부트

### 스프링 프레임워크란?
프레임워크 :  **어떤 규칙과 구조를 제공**하는 틀로, 개발자가 효율적으로 애플리케이션을 만들 수 있도록 돕는 도구
이미 정의된 구조와 기능을 제공해 개발자가 반복적인 작업을 줄이고 핵심 기능 구현에 집중할 수 있도록 도와줌 

- **3가지 핵심 특징**:
  1. **의존성 주입(DI)**: 객체 간 의존 관계를 프레임워크가 관리함.
  2. **제어의 역전(IoC)**: 객체의 생명 주기를 개발자가 아닌 프레임워크가 관리함.
  3. **관점지향 프로그래밍(AOP)**: 횡단 관심사를 분리하여 코드 중복을 줄임.

### 코드의 종류

1. **비즈니스 로직**:
   - 애플리케이션에서 **필수적**이고 **핵심적인 역할**을 하는 코드.
   - 필수 규칙에 따라 작성되어야 하며, 의존성이 강함.
   - 예: 주문 처리, 결제 계산 등.
2. **나머지 코드**:
   - **필수적이지만 핵심적이지 않은 코드**.
   - 예: 데이터베이스 저장/조회, 사용자 요청 처리 등.

- **스프링 프레임워크의 역할**:
  - 두 코드 종류를 분리하고, 비즈니스 로직을 더 쉽게 작성할 수 있도록 지원.

---

## 2. 스프링 부트란?

- **스프링 프레임워크 기반의 서버를 쉽게 만들 수 있게 해줌.**
- 자바로 서버를 만들려면 복잡하고 번거로운 작업이 많음. 이를 자동화한 것이 **스프링 부트**임.
- "오토 컨피규레이션" 기능으로 기본 설정 자동화.
  - 필요한 경우 개발자가 추가 설정 및 튜닝 가능.

### 주요 역할

- **웹 브라우저**를 통해 서버와 연결.
- 기본적으로 \*\*포트(예: 8080)\*\*를 통해 애플리케이션에 접근 가능.
- 하지만 HTTP 요청에 대한 응답이 없으면 에러 발생.\
  -> **컨트롤러**를 추가하거나 **정적 리소스**를 사용하여 해결 가능.

### 참고: 스프링 부트 공식 설명

> "스프링 부트는 실행만으로 독립적이고 기업 제품 수준의 애플리케이션을 만들 수 있음."

## 3. 컨트롤러와 정적 리소스

### 컨트롤러

- **HTTP 요청**을 받고, 이에 대한 응답을 처리하는 소스 코드.
- API 서버로 사용 가능하며, 데이터를 JSON 형태로 반환하는 경우가 많음.


### **API (Application Programming Interface)**
- **정의**: 프로그램이나 시스템 간에 **데이터를 주고받을 수 있는 규칙이나 방법**.
- **역할**: 클라이언트(사용자 측 프로그램)와 서버(데이터 제공자) 간의 **중간 다리** 역할.

### **API 서버**
- **정의**: API를 제공하는 **서버**.
- **역할**: 
  1. 클라이언트로부터 **요청**을 받고,
  2. 요청에 따라 데이터를 **제공**하거나 작업 결과를 **응답**.
- **특징**:
  - JSON과 같은 포맷으로 데이터 송수신을 주로 처리.
  - 다양한 클라이언트(웹, 앱 등)와 통신 가능.


### 정적 리소스

- 별도 로직 없이 파일을 그대로 전달.
- 예: HTML, CSS, JS 등의 정적 파일.

## 2. 주요 폴더 및 파일

### **`.idea`**
- **설명**: 인텔리제이에서 프로젝트 설정을 관리하는 폴더.
- **내용**: IDE에서 사용하는 설정 정보 저장.

### **`.mvn`**
- **설명**: 빌드 도구인 **메이븐(Maven)** 관련 파일.
- **내용**: 프로젝트 의존성 관리 및 빌드 프로세스 설정.

### **`src`**
- **설명**: 애플리케이션의 **소스코드**가 저장된 폴더.
- **내용**: Java 코드, 설정 파일, 리소스 등 포함.
- main / test 폴더로 나뉘고
- main : java폴더 : 자바 코드 , resources 폴더 : 정적리소스 폴더 

### target
- 메이븐이나 그래들 같은 빌드툴로 빌드했을때 서버에 배포할 수 있는 jar파일이 생성된다.
- .java 소스코드의 컴파일 파일인 .class 파일도 이곳에 생성됨
  
.java: 소스 코드. 개발자가 작성하고 수정하는 파일.
.class: 컴파일된 코드. JVM에서 실행되는 최종 파일.
JAR/WAR: 여러 .class 파일을 묶은 파일. 서버에 배포되는 형태.

.gitignore : git : 형상관리 툴, git 이 소스코드를 관리하다 보면 추적이 필요하지 않거나 무시하고 싶은 파일을 이 폴더에 저장해둠

pom.xml
project object model : 프로젝트 관련 정보를 하나의 개체로 모델링 하고 있다는 뜻.

### Annotation 

- **정의**: 
  - 애플리케이션 동작 및 설정 정보를 **선언적**으로 작성할 수 있는 기능.
- **예시**:  
  - `@SpringBootApplication`: 애플리케이션 진입점을 정의.
  -  자동설정을 활성화하며, 제거시 기본적인 설정을 로드하지 않아서 실행되지 않을 수 있음 
  - `@Controller`: HTTP 요청을 처리하는 컨트롤러 정의.
  - `@Service`: 비즈니스 로직을 처리하는 서비스 계층 선언.
  - 필요한 의존성을 객체로 만들어주는 것은 '컴포넌트 빈'으로 등록한다고 표현한다.
- **장점**:
 - 코드 간소화, 가독성 증가, 자동화 지원 

## 4. 참고 : URL 구조
- `http://localhost:8080/bye`
- `http://someuser:anypassword@localhost:8080/bye`

**구성 요소**:
- **scheme**: 프로토콜 (예: `http`, `https`)
- **user**: 사용자 이름 (옵션)
- **passwd**: 비밀번호 (옵션)
- **host**: 서버 주소 (예: `localhost`, `example.com`)
- **port**: 포트 번호 (기본적으로 HTTP는 80, HTTPS는 443이지만, 스프링 부트에서는 보통 8080 포트를 사용)
- **url-path**: 리소스 경로 (예: `/bye`, `/home`)


### 추가 요청 처리 흐름

1. **클라이언트 요청**
   - 클라이언트(예: 웹 브라우저)가 URL을 통해 요청을 보냅니다.
   - 예: `http://localhost:8080/`

2. **내장 서버(Tomcat, Jetty 등)**  
   - Spring Boot 애플리케이션에 내장된 서버가 클라이언트의 요청을 가장 먼저 받습니다.  
   - 서버는 이 요청을 처리하기 위해 Spring의 **DispatcherServlet**에 전달합니다.

3. **DispatcherServlet**  
   - 요청 경로(예: `/`)를 기준으로 어떤 **컨트롤러와 메서드**가 이 요청을 처리해야 하는지 찾습니다.  
   - 예: `/` 경로에 매핑된 메서드가 있다면, 해당 메서드로 요청을 전달합니다.

4. **컨트롤러(SimpleRestController)**  
   - 요청을 처리하기 위해 지정된 컨트롤러의 메서드가 실행됩니다.  
   - 컨트롤러는 클라이언트가 요청한 데이터를 생성하거나, 가공한 후 반환합니다.
   - 예: `"Hello"`

5. **응답 반환**
   - 컨트롤러의 반환값은 DispatcherServlet을 통해 클라이언트로 전달됩니다.  
   - 클라이언트는 브라우저나 HTTP 클라이언트를 통해 이 응답을 받습니다.
